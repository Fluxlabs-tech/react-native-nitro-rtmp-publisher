# react-native-nitro-rtmp-publisher

[![npm](https://img.shields.io/npm/v/react-native-nitro-rtmp-publisher.svg)](https://www.npmjs.com/package/react-native-nitro-rtmp-publisher)
[![license](https://img.shields.io/npm/l/react-native-nitro-rtmp-publisher.svg)](./LICENSE)
[![platform](https://img.shields.io/badge/platform-android-3DDC84.svg?logo=android)](#)

High-performance **RTMP camera publisher** for React Native, built on
[Nitro Modules](https://nitro.margelo.com) and
[RootEncoder](https://github.com/pedroSG94/RootEncoder).

- Native preview view (Camera2 + OpenGL ES, zero React Native UI thread cost)
- Hardware H.264 / H.265 / AV1 encoding via `MediaCodec`
- RTMP publish via RootEncoder's `RtmpCamera2`
- Adaptive bitrate, torch, zoom, exposure, auto-focus, stabilization, mirror
- Optional simultaneous local MP4 recording
- Type-safe API generated by Nitrogen — no JSON serialization on the hot path
- Per-second bitrate callback is **opt-in** (zero bridge cost otherwise)

> **Platform**: Android only at the moment. iOS contributions welcome.

---

## Install

```sh
npm install react-native-nitro-rtmp-publisher react-native-nitro-modules
# or
yarn add  react-native-nitro-rtmp-publisher react-native-nitro-modules
```

### Permissions (Android)

The library declares `CAMERA`, `RECORD_AUDIO`, `INTERNET`, `ACCESS_NETWORK_STATE`
and `WAKE_LOCK` in its own manifest — manifest merging picks these up
automatically, so you don't have to repeat them.

You still need to **request `CAMERA` and `RECORD_AUDIO` at runtime** via
`PermissionsAndroid` (or `expo-camera` / `expo-av` permission helpers) before
mounting the view — runtime grants aren't covered by manifest merging.

## Minimal usage

```tsx
import { useMemo, useRef } from 'react'
import { StyleSheet, Button, View } from 'react-native'
import { callback } from 'react-native-nitro-modules'
import {
  RtmpPublisherView,
  type RtmpPublisherViewMethods,
  type RtmpConnectionEvent,
} from 'react-native-nitro-rtmp-publisher'

export default function App() {
  const publisher = useRef<RtmpPublisherViewMethods | null>(null)

  const hybridRef = useMemo(
    () => callback((ref: RtmpPublisherViewMethods) => {
      publisher.current = ref

      const rotation = ref.getCameraOrientation()
      ref.prepareVideo(1280, 720, 30, 2_500_000, 2, rotation)
      ref.prepareAudio(128_000, 44_100, true)
      ref.startPreview('back', 1280, 720)

      ref.setOnConnectionEvent((event, msg) => {
        console.log('rtmp:', event, msg)
      })
    }),
    []
  )

  return (
    <View style={{ flex: 1 }}>
      <RtmpPublisherView
        style={StyleSheet.absoluteFill}
        forceHardwareCodec={true}
        videoCodec="h264"
        audioCodec="aac"
        aspectRatioMode="adjust"
        mirrorPreview={false}
        mirrorStream={false}
        hybridRef={hybridRef}
      />
      <Button title="Go live" onPress={() =>
        publisher.current?.startStream('rtmp://your-ingest/app/streamKey')
      } />
    </View>
  )
}
```

> **Important** — wrap the `hybridRef` callback in `useMemo([], …)`. If the
> callback identity changes between renders, the view re-initializes on
> every render and the camera will thrash open/close.

## Props

All props are required (set them once in JSX).

| Prop                  | Type                                  | Default     | Description |
|-----------------------|---------------------------------------|-------------|-------------|
| `forceHardwareCodec`  | `boolean`                             | `true`      | Pin both MediaCodec encoders to HARDWARE. Set `false` to allow SW fallback. |
| `videoCodec`          | `'h264' \| 'h265' \| 'av1'`           | `'h264'`    | Encoder. Most RTMP ingest expects H.264. |
| `audioCodec`          | `'aac' \| 'g711' \| 'opus'`           | `'aac'`     | Audio encoder. RTMP standard is AAC. |
| `aspectRatioMode`     | `'fill' \| 'adjust' \| 'none'`        | `'adjust'`  | How the preview maps onto the view bounds. |
| `mirrorPreview`       | `boolean`                             | `false`     | Flip on-screen preview horizontally. Stream unaffected. |
| `mirrorStream`        | `boolean`                             | `false`     | Flip encoded stream horizontally. Preview unaffected. |
| `thermalWarningThreshold` | `ThermalStatus`                   | `'severe'`  | Min OS thermal level that fires `onThermalWarning`. `'none'` disables. |
| `audioSource`         | `'mic' \| 'camcorder' \| 'voiceRecognition' \| 'voiceCommunication' \| 'unprocessed'` | `'camcorder'` | `MediaRecorder.AudioSource`. `'camcorder'` is the right default for streaming — less aggressive AGC than `'mic'`. |
| `autoRotateStream`    | `boolean`                             | `true`      | Library wires an internal `OrientationEventListener` to `setStreamRotation`. Disable for manual control. |
| `streamMode`          | `'lowLatency' \| 'balanced' \| 'quality'` | `'balanced'` | Pipeline preset. Use `'quality'` for 3hr+ broadcasts — forces monotonic timestamps to avoid A/V drift. |

## Methods

Call via `hybridRef`. The full ref type is `RtmpPublisherViewMethods`.

### Lifecycle

| Method | Notes |
|---|---|
| `prepareVideo(width, height, fps, bitrate, iFrameInterval, rotation): boolean` | Configure video encoder. Call **before** `startPreview`. `rotation` should usually be `getCameraOrientation()`. |
| `prepareAudio(bitrate, sampleRate, isStereo): boolean` | Configure audio encoder. Call **before** `startPreview`. |
| `startPreview(facing, width, height): void` | Open camera, render frames into the view. |
| `stopPreview(): void` | Release camera. |
| `startStream(url): void` | Begin RTMP publish. Queues until the GL surface is ready, so calling it immediately after `prepareVideo`/`prepareAudio` is safe. |
| `stopStream(): void` | End RTMP publish. |
| `setAuthorization(user, password): void` | Set AMF auth credentials. Call before `startStream` for ingests that require it. |
| `setStreamRotation(rotation): void` | Update encoder rotation mid-session (0/90/180/270). Useful with `OrientationEventListener`. |
| `requestKeyFrame(): void` | Force the next encoded frame to be an IDR. No-op when not streaming. |

### Status

| Method | Returns |
|---|---|
| `isStreaming()` | `boolean` |
| `isOnPreview()` | `boolean` |
| `getCameraOrientation()` | Recommended rotation (degrees) for `prepareVideo`. |
| `getStreamWidth()` / `getStreamHeight()` | Currently configured encoder resolution. |
| `getCurrentBitrate()` | Configured **target** bitrate (bps). For the measured TX rate, subscribe to `setOnBitrateChange`. |

### Adaptive bitrate

**Built-in adaptive bitrate.** The library samples the measured TX bitrate
every second and adjusts the encoder via `setVideoBitrateOnFly`:
- decreases when the RTMP send-buffer fills up (network congestion),
- slowly recovers toward `maxBitrate` when there's headroom.

```ts
// Cap at 2.5 Mbps. On congestion drop 20% of current bitrate; on recovery
// raise 5% per second.
ref.setAdaptiveBitrate(2_500_000, 20, 5)

// Disable
ref.setAdaptiveBitrate(0, 0, 0)
```

`maxBitrate` should match (or be slightly above) the `bitrate` you passed to
`prepareVideo`. The adapter resets to the ceiling at every fresh `startStream`.

If you want full manual control instead, subscribe to `setOnBitrateChange`
and call `setVideoBitrateOnFly(...)` yourself:

```ts
ref.setOnBitrateChange((txBps) => {
  if (txBps < SOME_THRESHOLD) ref.setVideoBitrateOnFly(LOW)
  else                        ref.setVideoBitrateOnFly(HIGH)
})
```

### Auto-reconnect

Mobile networks drop. By default the library does NOT retry on its own — opt
in with `setAutoReconnect(maxAttempts, backoffMs)`:

```ts
// On every `connectionFailed`/`disconnect`, retry up to 5 times with a
// 3-second backoff between attempts. Fires a `reconnecting` event right
// before each retry; if the budget runs out you'll get a final `disconnect`.
ref.setAutoReconnect(5, 3000)
```

The retry budget is re-seeded on every fresh `startStream(url)`. Calling
`stopStream()` aborts any in-flight retry.

If you want to drive retries yourself instead, use the primitives directly:

```ts
ref.setReTries(5)                      // budget for the session
const ok = ref.reTry(3000, 'manual')   // returns false if budget exhausted
```

### Local recording

Independent of streaming — can record while live, or without ever streaming.

```ts
const started = ref.startRecord('/storage/emulated/0/Movies/clip.mp4')
// returns false if the path is invalid or the recorder is already running
ref.stopRecord()
// also: pauseRecord(), resumeRecord(), getRecordStatus()

// Opt-in: receive status transitions (started/recording/paused/resumed/stopped)
ref.setOnRecordStatusChange((status) => console.log('record:', status))
```

### Long-form / 3-hour streams

For broadcasts that run for hours, A/V can drift if MediaCodec's encode clock
and the audio capture clock disagree. Set the `streamMode="quality"` prop:

```tsx
<RtmpPublisherView streamMode="quality" ... />
```

That preset:
- Forces **monotonic incremental timestamps** (`forceIncrementalTs`) — prevents
  audio/video PTS divergence over time
- Uses `TimestampMode.CLOCK` (system monotonic clock) instead of MediaCodec
  buffer PTS
- Enlarges the RTMP send cache to ~4s and writes larger chunks (smoother under
  link variance)
- Softens the adaptive-bitrate exponential factor so bitrate doesn't oscillate

If you need finer control, call the primitives directly:

```ts
ref.forceIncrementalTs(true)
ref.setStreamDelay(50)        // +50ms audio delay to compensate for AEC echo
```

### FPS lock in low light

By default Camera2's auto-exposure can extend frame duration to brighten
dark frames — your "30fps" stream becomes 15fps once the sun goes down.
Lock the AE FPS range:

```ts
ref.setForceFpsLimit(true)
```

Trade-off: darker night frames, but smooth motion. Recommended on for any
sports / motion-heavy stream.

### Audio source for streaming

The default `MediaRecorder.AudioSource.MIC` is tuned for phone calls — AGC
crushes dynamic range, noise gate kills ambient sound. Pick a better source
via the `audioSource` prop:

| Source | Use for |
|---|---|
| `'camcorder'` | **Default.** Streaming, vlogs, anywhere voice + room mix |
| `'mic'` | Quiet rooms, talking heads, when AGC actually helps |
| `'voiceRecognition'` | Raw signal, you handle mixing |
| `'voiceCommunication'` | Two-way streams with echo cancellation |
| `'unprocessed'` | Music capture (API 24+) |

Single biggest perceived audio-quality win available.

### Auto rotation

The library attaches an internal `OrientationEventListener` and snaps the
encoder rotation to nearest 90° on physical device rotation. Enabled by
default via `autoRotateStream={true}`. Disable if you want to control
rotation yourself (e.g. locked landscape).

### Foreground service

Long streams should survive brief backgrounding / lock-screen. Opt in by
setting the `foregroundServiceTitle` prop — the library auto-starts the
service on `startStream` and auto-stops it on `stopStream` / view drop:

```tsx
<RtmpPublisherView
  foregroundServiceTitle="Live stream"
  foregroundServiceText="Broadcasting"
  ...
/>
```

Leave both empty (the default) for camera apps where the user is always
looking at the preview — no notification, no service.

The service:
- Presents a persistent low-priority notification (channel `rtmp_publisher_channel`)
- Declares `foregroundServiceType="camera|microphone"` (required Android 14+)
- Cleans up automatically — JS never has to call `start`/`stop`

Manifest entries and `FOREGROUND_SERVICE_*` permissions are declared in the
library; manifest merging picks them up.

> Caveat: even with the FG service, Android destroys the `SurfaceView`
> when the app is fully backgrounded, so the camera capture stops on
> `surfaceDestroyed`. The FG service buys you protection from process
> death (quick resume on return) but not true "phone in pocket, still
> streaming" behaviour — that requires rendering to an offscreen surface,
> which is a separate body of work.

### Thermals

Sustained streaming can throttle the SoC and tank the encoder. The library
hooks Android 10+'s `PowerManager.OnThermalStatusChangedListener` so you can
react without polling.

```ts
// Set the trip level via the `thermalWarningThreshold` prop (default 'severe').
// Subscribe to receive callbacks. The listener is only registered when you
// actually subscribe — zero PowerManager touch otherwise.
ref.setOnThermalWarning((status) => {
  // 'none' | 'light' | 'moderate' | 'severe' | 'critical' | 'emergency' | 'shutdown'
  if (status === 'severe') ref.setVideoBitrateOnFly(1_000_000)
  if (status === 'critical') ref.stopStream()  // your call
})

// One-shot read
const now = ref.getThermalStatus()
```

The callback fires twice for a typical heat event:
1. When the state first crosses the threshold (entering the warning zone)
2. Once when it falls back below the threshold (clearing — useful for "all
   good, dial back up" logic)

Requires Android 10 (API 29). On older devices `getThermalStatus()` returns
`'none'` and the callback never fires.

#### Testing thermal handling without actually overheating

```bash
adb shell cmd thermalservice override-status 3   # SEVERE
adb shell cmd thermalservice override-status 0   # back to normal
adb shell cmd thermalservice reset               # release override
adb shell dumpsys thermalservice                 # inspect current state
```

## Performance notes

Everything in the hot path stays native:

- **Camera2 capture → SurfaceTexture → OpenGL render → MediaCodec input
  Surface** — never crosses into JS, no per-frame allocation.
- **MediaCodec H.264/AAC encode** — hardware-accelerated (pinned with
  `forceHardwareCodec={true}`). Effectively zero CPU.
- **FLV muxing + RTMP TX** — RootEncoder's `RtmpClient` on a background thread.
- **JS bridge** — touched only on:
  - `setOnConnectionEvent` transitions (rare),
  - `setOnBitrateChange` (opt-in, ~1 Hz when subscribed),
  - any imperative method you call.

Knobs that move CPU/GPU usage:

| Lever | Effect |
|---|---|
| Resolution (1080p → 720p) | ~2× less work end-to-end |
| FPS (30 → 24) | -20% |
| `forceHardwareCodec={true}` | Required for "free" encode |
| Don't subscribe to bitrate | Saves one bridge round-trip per second |
| Don't record while streaming | Avoids second encoder |

The `setVideoBitrateOnFly(bitrate)` method lets you implement adaptive bitrate
on top of network-quality signals without ever resetting the encoder pipeline.

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                            JavaScript                            │
│   <RtmpPublisherView                                             │
│     forceHardwareCodec videoCodec mirrorPreview … hybridRef={…}/>│
│                       ▲                                          │
│                       │ generated by Nitrogen                   │
│                       ▼                                          │
│   ref.prepareVideo() / startStream() / setZoom() / …             │
└─────────────────────────────────────────────────────────────────┘
                          │ JSI (one call, no JSON)
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│            HybridRtmpPublisherView.kt  (this package)            │
│                                                                  │
│    OpenGlView ──── RtmpCamera2 ──── ConnectChecker               │
│         │              │                  │                      │
│         ▼              ▼                  ▼                      │
│   Surface preview   MediaCodec       RTMP TX over TCP           │
│                     (HW encode)                                  │
└─────────────────────────────────────────────────────────────────┘
                          │
                          ▼
                  RootEncoder runtime
```

- **JS surface** — typed spec at `src/specs/RtmpPublisher.nitro.ts`. Nitrogen
  generates the C++ shim, the Kotlin abstract base, and the Fabric view
  component config.
- **Native implementation** — `HybridRtmpPublisherView` extends the generated
  spec; one class owns the `OpenGlView` + `RtmpCamera2`.
- **Lifecycle** — `surfaceDestroyed` stops both stream and preview so the
  camera releases when the view unmounts. When the surface is destroyed mid-stream
  (rotation, backgrounding) a synthetic `disconnect` event is emitted to JS so
  the UI state stays consistent.
- **Wake-lock** — a `PARTIAL_WAKE_LOCK` is acquired on `startStream` and released
  on `stopStream`/`disconnect`; `keepScreenOn` is toggled on the preview view
  for the same window.

## Common pitfalls

| Symptom | Cause | Fix |
|---|---|---|
| Camera opens/closes in a loop | `hybridRef` callback recreated per render | Wrap in `useMemo([], …)` |
| Output stream lands in landscape after a few seconds | `prepareVideo` called after `startPreview`; encoder rotation latched late | Call `prepareVideo` + `prepareAudio` **before** `startPreview` |
| Output is upside down | Rotation applied twice | Pick one source of truth: either pass `rotation` to `prepareVideo` OR drive `setStreamRotation` from an `OrientationEventListener`, not both. |
| `ViewManagerPropertyUpdater: Could not find generated setter` warning | RN looks for legacy `@ReactProp` setters; Nitro uses Fabric state instead | Harmless — props still propagate |
| `Unable to resolve module ./index` from Metro | The package symlink resolves recursively into `example/` | Use `metro.config.js` with a `blockList` regex for that path |
